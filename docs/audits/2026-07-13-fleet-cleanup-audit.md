## What this repo is

`menno420/sim-lab` is the fleet's **evidence seat**. Ideas routed from
`menno420/idea-engine`'s `control/outbox.md` (`status: sim-ready`) are settled
here with reproduced facts — numeric simulation, measured prototype/spike, or
(only when neither applies) a structured `JUDGMENT-ONLY` analysis — via a
five-question validity gate (comparable / uncorrupted / robust / reproducible
/ limits). The output is one finalized **verdict** (`approve` /
`reject` / `needs-more-evidence`) appended to `control/outbox.md`, addressed
to the fleet manager for routing. The seat does not build products and its
only writable repo is itself (`README.md`, Q-0260).

## CI setup and health

Two workflows, both kit-owned and regenerated by `bootstrap.py adopt/upgrade`:

- **`substrate-gate.yml`** — the required check. A `control/**`-only diff
  short-circuits to a fast lane (`bootstrap.py check --strict --status-only`);
  everything else runs the full hygiene gate, keyed to whichever
  `.sessions/*.md` card the diff actually touches (not the newest-by-mtime
  card, to defeat a documented CI-mtime-flattening false-green class).
- **`auto-merge-enabler.yml`** — arms native GitHub squash auto-merge on
  every non-draft PR whose head ref matches an explicit prefix allowlist
  (`substrate.config.json` → `automerge.branch_patterns`), refusing to arm if
  the base branch requires zero status-check contexts, re-reading the
  `do-not-automerge` label fresh after a 15 s grace window, and — the
  repo-specific customization — skipping the arm while the PR's own in-diff
  session card is still `in-progress` (the "born-red" card convention), so a
  green intermediate commit can never squash-merge a multi-commit session
  early.

**Verified live**: PR #115 (the most recent merge, `kit: upgrade
substrate-kit v1.7.0 → v1.15.0`) shows both checks (`substrate-gate`,
`enable-auto-merge`) `completed`/`success`, each finishing in under 20

## Suggestions

1. **A cheap fleet-wide checker for the "config says X, prose heartbeat says
   Y" class** (Inconsistency #1). Since `substrate.config.json` is a
   kit-owned, machine-readable field and `kit: vX.Y.Z` in `control/status.md`
   / `docs/current-state.md` is free text a human (or session) types by hand,
   a one-line `bootstrap.py check` rule that warns (never reds) when the two
   disagree by more than one PR-cycle would catch this drift class across
   every repo running substrate-kit, not just this one — it costs one check
   in the kit and pays off fleet-wide.

2. **`control/outbox.md` is 834 KB / 1,068 lines and growing by roughly one
   `## VERDICT` block (15–25 lines) every 10–25 minutes tonight.** The
   repo's own ORDER 005 item 5 already flags this as heading toward the same
   ~256 KB practical wall that `idea-engine`'s `control/outbox.md` hit (per
   its ASK 004), and is explicitly waiting on a fleet-manager rollover/
   archival convention before restructuring. Worth centralizing: this is a
   second lane hitting an identical append-only-ledger-size wall, which
   argues for solving it once at the substrate-kit level (a documented
   rollover convention — e.g. `control/outbox.md` → `control/outbox-archive/
   <NNN>.md` at a size or entry-count threshold) rather than each lane
   improvising its own answer when it gets there.

3. **The `review-queue.md` escape valve (Inconsistency #2) is worth a
   deliberate yes/no**, not indefinite silence: either retire the convention
   as unneeded (the self-check/dual-arm/git-trail discipline in every
   verdict may already substitute for a human second look) or find out why
   it's never fired despite 115 PRs. Either answer is useful and cheap to
   get via the existing `docs/retro/questions.md` mechanism.

4. **For future fleet-wide EAP-style sweeps**, a seat this tightly looped
   (a merge every 10–25 minutes, sustained for hours, 0 open PRs by design)
   has essentially no actionable surface for an external audit beyond
   documentation review — the loop's own `swept @ <sha>` self-report
   (`control/inbox.md` ORDER 005) is a reliable enough freshness signal on
   its own. A future sweep could budget less time on repos whose own
   heartbeat is this fresh (verify the "0 open PRs" claim, confirm CI is
   green, stop) and reserve more time for repos that are actually dormant or
   drifted — this repo was neither.
