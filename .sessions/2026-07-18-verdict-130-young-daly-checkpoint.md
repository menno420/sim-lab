# VERDICT 130 — the checkpoint-interval optimum: a long fleet job that can crash and restart from its last checkpoint faces TWO competing wastes — checkpoint OVERHEAD (each checkpoint costs C, taken every τ of work, ~C/τ per unit useful work) and lost RECOMPUTE work on a crash (a Poisson failure at rate 1/M lands on average halfway through the current interval, discarding the work since the last checkpoint, ~τ/(2M) per unit work) — and their sum C/τ + τ/(2M) is a CONVEX falling+rising pair, so total wasted wall time is U-SHAPED in the checkpoint interval τ with a unique INTERIOR optimum where the two wastes are equal: τ*=√(2·C·M) (Young 1974 first-order), overhead floor √(2C/M) (Daly 2006 exact); intuition says "to be safe against failures, checkpoint as often as you can afford" but that walks UP the left arm of the U (the C/τ term diverges as τ→0), so over-checkpointing is NOT "safer", it is exactly as wasteful in leading order as the symmetric under-checkpointing (τ*·k and τ*/k inflate the total overhead by the same factor ½(k+1/k)); and the optimum scales only as √M — a 4× more reliable machine (M→4M) warrants an interval only 2× longer and HALVES the achievable overhead floor √(2C/M) — so the transferable correction for any interruptible process that persists progress against a snapshot and pays a snapshot cost (a training run, a batch pipeline, a long agent task, a stream processor, a simulation restart-dump) is that the checkpoint/snapshot/commit interval is a U-shaped OPTIMISATION, not a one-sided safety dial: set it to τ*=√(2·C·MTBF), read the overhead floor as √(2C/MTBF), and since the optimum grows only as √MTBF, spend engineering effort on shrinking the checkpoint cost C (incremental/async checkpoints) rather than on cranking the frequency;
