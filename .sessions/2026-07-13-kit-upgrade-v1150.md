## 💡 Session idea

The carve-out scanner names host-added STEPS but is blind to host edits
INSIDE a kit-named step — and that asymmetry nearly lost a behavior
silently. This upgrade's scan reported exactly one carve-out ('Skip
arming while the PR's own in-diff session card is in-progress') yet the
pre-regen enabler carried a SECOND host customization living inside the
kit's own 'Enable native auto-merge (squash)' step: the `--body
"Head-ref: $HEAD_REF"` squash-provenance stamp (banked file ~199–205).
It survived only because the full pre-regen file gets banked — the report
never named it, so a reader trusting the ⚠️ section's enumeration would
conclude one carve-out existed, not two. The fix is cheap and structural:
the scanner already matches steps by name to detect additions, so for
steps whose names MATCH the kit template it should also compare the step
body (hash or line-diff) and emit a `carve-out (in-step edit): <step
name> — body differs from template@old` line per mismatch. That turns
"the bank is your only witness" into an enumerated, greppable report
line, at the cost of one extra comparison in a loop the scanner already
runs. Evidence anchors: `.substrate/upgrade-report.md` § "⚠️ Gate
carve-outs" (one line where two customizations existed) vs
`.substrate/backup/auto-merge-enabler.pre-regen-189497a9.yml` lines 98
and 199–205.
