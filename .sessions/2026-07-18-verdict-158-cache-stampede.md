# VERDICT 158 — cache-stampede / request-coalescing herd law

Reproduce PROPOSAL 145 (round-34 FLEET slot, P145 → V158, +13): the cache-stampede / request-coalescing herd law. Folk belief says a cache with a near-100% hit ratio protects the origin and a HOTTER key (higher request rate) is SAFER because it is almost always served from cache — so to tame origin load you raise the hit ratio / lengthen the TTL. Reality (cache stampede / dogpile / thundering herd): caching a hot key does NOT reduce its worst-case origin recompute load — it CONCENTRATES that load into a herd, and the herd grows LINEARLY with request rate. When a hot key's cached value expires it is stale for the length of one recompute T, and under a plain TTL cache WITHOUT coordination EVERY request that arrives during that window independently recomputes the same value at the origin. Model one expiry event as a window [0, T): requests arrive as a Poisson process of rate lambda, so the number that pile onto the origin is exactly Poisson(lambda·T) — expected herd E = lambda·T (the Wikipedia cache-stampede example verbatim: 10 req/s with a 3 s render → 30 processes recomputing the same page at once). The herd therefore SCALES with the request rate: doubling traffic doubles the per-expiry origin burst even as the cache hit ratio climbs toward 100%. Request coalescing (single-flight / dogpile lock) lets the FIRST request in the window recompute and makes every other request wait on that in-flight result, collapsing the herd to exactly 1 recompute per expiry (E = 1 − e^{−lambda·T}, ~1 for a hot key) — INDEPENDENT of lambda. So the cure for a stampede is coalescing, not a longer TTL (a longer TTL leaves the per-expiry herd size lambda·T untouched). The verifier draws the Poisson arrival process on [0, T) by summing interarrival ~ Exp(lambda) until the clock passes T, over R=4000 independent replications per config, and evaluates three ordered gates: G1 CFG_LOW verifier-correctness (naive herd matches the EXACT Poisson mean lambda·T=5, |z|<3) → G2 CFG_HIGH verifier-correctness (naive herd matches the EXACT Poisson mean lambda·T=30, |z|<3 — the herd is Poisson(lambda·T) and SCALES 6× with 6× the rate) → G3 CFG_HEAD headline (the paired amplification naive_herd − coalesced_recomputes per expiry sits z≥3 ABOVE zero — coalescing removes ~lambda·T−1 redundant recomputes per expiry, a ~30×→1× collapse). On the pinned world (SEED=20260717, T_RECOMPUTE=1.0, SIGMA_GATE=3.0) all three gates hold in order and the run is deterministic. The trap: a practitioner who raises the hit ratio / lengthens the TTL to protect the origin from a hot key is treating the wrong variable — the per-expiry herd size is lambda·T regardless of hit ratio, and only coalescing (single-flight) collapses it.

## 💡 Session idea
This slice pins the stampede at ONE recompute window length (T=1.0) and contrasts exactly TWO regimes per expiry — the plain-TTL herd (Poisson(lambda·T)) vs the fully-coalesced 1-recompute collapse — at two rates (lambda=5, 30). A genuinely distinct NEXT FLEET-slot slice, distinct from this all-or-nothing coalescing head, pins the **PARTIAL-coalescing / probabilistic-early-recompute regime**: real dogpile mitigations are rarely a perfect single-flight lock — they are (i) a lock with a bounded wait window so a fraction p of the herd still recomputes when the lock is contended, or (ii) XFetch-style probabilistic early expiration where each request independently recomputes with a small probability that rises as the TTL nears expiry, spreading recomputes BEFORE the hard expiry instead of piling them after. The pre-registrable structure: (G-a) confirm the plain-TTL herd is Poisson(lambda·T) by ≥3σ (the uncoordinated baseline, as here); (G-b) a monotonicity headline — confirm the expected origin recompute count is strictly INCREASING in the coalescing-miss probability p by ≥3σ, interpolating between the coalesced ~1 (p→0) and the full herd lambda·T (p→1), so partial coalescing buys a proportional-not-binary reduction; (G-c) an early-expiration control — confirm probabilistic early recompute strictly LOWERS the peak per-instant origin concurrency vs a hard-expiry herd by ≥3σ at equal total recompute budget (spreading beats collapsing when a full lock is unavailable). The operator lesson sharpens from "coalescing collapses the herd to 1" to "the origin-load reduction is set by how completely you coalesce — a leaky lock leaves a herd proportional to its miss rate, and early probabilistic expiration trades a taller post-expiry spike for a flatter pre-expiry spread." A seed-pinnable queueing mechanism deliberately distinct from this binary-coalescing head, converting the current all-or-nothing collapse into a first-class question (how much does PARTIAL coalescing actually buy?) with an exact reference (Poisson thinning) at every miss-probability grid point.
