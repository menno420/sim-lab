# VERDICT 003 В· INTAKE 002 вЂ” Discord OAuth trust-gate (per-server personal stats)

**Phases 1вЂ“2 carry NO auth surface and do NOT wait on this verdict** (required statement, source В§3): the story page (static, zero auth/backend) and the data explorer (versioned static JSON + client-side search, no live DB, no auth) are shippable now. ## METHOD LABEL: `prototype + JUDGMENT-ONLY` (mixed вЂ” ladder rungs 2 and 3)
The executable core is a **measured spike** (rung 2): the OAuth authorization-code trust-gate is fully modelable as a state machine, so `oauth_trust_gate_sim.py` builds a reference relying-party implementing each of В§5's six controls and runs **13 executed adversarial attacks** against it, seeded (5 seeds) and rate-limit swept (3Г—3). The threats that **cannot** be exercised from this environment вЂ” live Discord IdP enforcement, TLS/MITM, Railway hosting/secret posture, pre-auth IP-keyed limiting, and the **unbuilt** В§4 API вЂ” are done as **JUDGMENT-ONLY** structured analysis (rung 3), labelled as such and listed as launch-time live tests. The label reflects the mix; in the outbox grammar it fills `evidence: prototype + JUDGMENT-ONLY`.

## What the spike MEASURES
A modelled Discord authorization server (`ModelIdP`: authorization-code + PKCE S256, single-use codes, exact-match redirect allow-list, scope-bound tokens) + the relying party (`StatsSite`, the reference implementation of В§5) + an attacker. Each В§5 control is a concrete mechanism, then attacked:
- **state store** вЂ” server-side, bound to the browser session, single-use (CSRF/state).
- **PKCE S256** вЂ” code bound to a challenge; wrong verifier rejected (code injection defense-in-depth).
- **server-authoritative minimal scopes** + **fail-closed** grant check (`identify`+`guilds` exactly).
- **token discarded** вЂ” the authed session stores only `{user_id, guild_ids, issued_at}`; no access/refresh token, no `offline_access`.
- **IDOR-by-session-identity** вЂ” a client-supplied `user_id` is ignored; only guilds in the session set are readable.
- **token-bucket rate limiter** вЂ” per identity+endpoint, swept over capacityГ—refill.

## What it SETTLED (the load-bearing claims вЂ” executed)
1. **All 13 modelled attacks are DEFEATED** by the reference implementation, identically across all 5 seeds. Per В§5 item:
   - **В§5.1 scope minimalism** вЂ” the site requests exactly `{identify, guilds}` (server-authoritative); an over-scoped grant from a malicious/misconfigured IdP is **rejected fail-closed** (`scope_violation`); a tampered `redirect_uri` is rejected by the exact-match allow-list. **PASS.**
   - **В§5.2 no long-lived token storage** вЂ” after login the session holds only `{user_id, guild_ids, issued_at}`; a full session-store dump recovers **zero** Discord credentials; no refresh token is requested. **PASS.**
   - **В§5.3 CSRF/state** вЂ” forged state, replayed (consumed) state, and a login-CSRF/session-fixation attempt (attacker's code+state completed in the victim's browser) are all rejected; an honest retry after a forged attempt still works. **PASS.**
   - **В§5.4 code replay/injection** вЂ” a used authorization code is rejected on reuse (single-use); PKCE rejects a wrong verifier. **PASS.**
   - **В§5.5 per-user isolation** вЂ” a spoofed `user_id` param is ignored (identity = session); a cross-server read (a guild the user is not in) is rejected (`forbidden_guild`). **PASS.**
   - **В§5.6 rate limiting** вЂ” a burst is bounded to bucket capacity across the full 3Г—3 sweep; refill restores access. **PASS.**
   - **В§5 abuse-case walkthrough** вЂ” this executed suite **is** the walkthrough. **PASS.**
2. **Two HOLES surface in the terse design** (headlines, reproduced every seed):
   - **HOLE-1 stale guild membership** вЂ” the session caches the guild set at login, so a user who **leaves** a server can still read its stats until the session expires. Fix: re-validate membership per request, or cap session TTL short.
   - **HOLE-2 `guilds` over-read** вЂ” `identify`+`guilds` returns the user's **entire** guild list when only the currently-viewed server is needed. Privacy over-collection. Fix: check membership for only the viewed server; do not retain the full list.
3. **Phases 1вЂ“2 need no auth** and are decoupled вЂ” shippable regardless of this verdict (source В§3).
